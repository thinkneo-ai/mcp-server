# Incident — Hardcoded Production Keys in Public Tests

**Date:** 2026-04-24 21:41 UTC (commit), detected 2026-04-25 via GitGuardian
**Severity:** HIGH (production credentials exposed in public repo)
**Status:** Resolved 2026-04-25 ~05:00 UTC
**Customer impact:** None detected (no unauthorized usage of leaked keys observed in audit logs)

## Timeline

- 24/abr 21:41 — Hardcoded Resend API key + master API key committed in test fixtures (commits 89225c2+)
- 25/abr ~04:30 — GitGuardian alert email received
- 25/abr 04:45 — Resend key rotated by Fabio
- 25/abr 05:00 — Master API key rotated by Fabio
- 25/abr 05:00 — Test fixtures replaced with env vars (THINKNEO_TEST_API_KEY) + fake placeholders
- 25/abr 05:05 — GitHub secret added, CI green confirmed (10/10 gates)
- 25/abr 05:10 — DB password default removed from database.py
- 25/abr 05:15 — Gitleaks pre-commit hook + GitHub Action added

## Root Cause

During Feature 5 (Functional Tests, abr 24), test fixtures were created using real production credentials copied from local environment, instead of synthetic fake strings. Four test files affected:

- `tests/unit/test_log_redaction.py` (Resend API key as test data)
- `tests/functional/test_every_mcp_tool.py` (master API key hardcoded)
- `tests/functional/test_every_a2a_skill.py` (master API key hardcoded)
- `tests/functional/test_a2a_governance_e2e.py` (master API key hardcoded)

No pre-commit hook was in place to detect secrets before push.

## Mitigation Applied

1. Both keys rotated in production (Resend + MCP master key)
2. All test files updated to use `os.environ["THINKNEO_TEST_API_KEY"]`
3. Synthetic placeholders used where real values were previously embedded (`re_FAKE_TEST_KEY_NOT_REAL_*`)
4. GitHub Actions secret `THINKNEO_TEST_API_KEY` configured
5. Container restarted, health verified (200 OK)
6. DB password default removed from `database.py`

## Audit of Exposure Window

**Window:** 2026-04-24 21:41 UTC → 2026-04-25 05:00 UTC (~7 hours)
**Repo visibility:** PUBLIC during entire window

**Findings:**
- 39 total requests during window
- 12 anonymous (public tools, no auth key used)
- 27 using the master key hash (`8bd263da...`) — **all from our own CI functional tests** at 03:28-03:29 UTC
- Tool calls match exactly the test files: `a2a_set_policy`, `bridge_mcp_to_a2a`, `a2a_log`, `start_trace`, `register_claim`, etc.
- 22 unique IPs during window — all identified as Azure CI runners (20.x, 40.x, 52.x), DigitalOcean (143.198), known crawlers (YellowMCP 187.77, Thailand 58.10), AWS (54.67)
- **Zero unknown IPs. Zero anomalous request patterns.**

**Conclusion: No unauthorized usage of leaked keys detected.** All requests during the exposure window are attributable to our own automated tests and known crawlers. No breach notification required.

## Lessons Learned

1. Real credentials must NEVER be used as test fixtures, even temporarily
2. Synthetic strings are mandatory; convention: `FAKE_`/`TEST_`/`NOT_REAL_` prefixes obvious to automated scanners
3. Pre-commit hooks are mandatory infrastructure, not optional
4. CI must include secret scanning as blocking gate
5. Speed of delivery never justifies security shortcuts

## Prevention Implemented

- Pre-commit hook: `.pre-commit-config.yaml` with gitleaks (commit `1766e70`)
- GitHub Action: `.github/workflows/gitleaks.yml` — server-side secret detection on every push/PR (commit `1766e70`)
- CONTRIBUTING.md: security guidelines for test fixture creation
- tests/README.md: env var documentation for test keys

## Cost

- Direct cost: $0 (no abuse detected, no breach notification required)
- Indirect cost: ~2 hours incident response, 2 keys rotated, container restart
- Reputational: GitGuardian alert on public repo (visible to potential contributors)
